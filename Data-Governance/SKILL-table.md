# 数据治理 · 表级 DDL 规范检查

## 概述

本 skill 专门对 Hive 数仓建表语句的 **表级别** 规范性进行审查，不涉及字段层面的检查。
覆盖以下五个维度：

1. **分层前缀合法性**（L-01 ～ L-03）
2. **各层命名结构**（S-ODS / S-DIM / S-DWD / S-DWS / S-ADS）
3. **字符规范**（C-01 ～ C-07）
4. **语义清晰度**（M-01 ～ M-04）
5. **表注释规范**（T-01 ～ T-07）
6. **模型冗余**（R-01 ～ R-03）

输入为包含一个或多个 `CREATE TABLE` 语句的 DDL 文档。
输出为 **Excel 文件**，包含「表级问题」sheet 页。

> **边界说明**：本 skill 仅做表级检查。字段级检查请使用配套的「数据治理-字段级DDL规范检查」skill。

---

## 前置要求

收到输入后，无需提前询问，直接开始解析和检查。以下两种情况例外，需先向用户确认：

- **无法判断所属层次**：表名前缀不在 `ods/dim/dwd/dws/ads` 范围内，需询问该表属于哪一层或是否为豁免表。
- **特殊豁免**：用户主动说明某些表为历史遗留表或外部对接表，可降级处理，记录豁免原因。

## 规则库

### 维度一：分层前缀

| 规则 ID | 规则描述 | 严重级别 |
|--------|---------|---------|
| L-01 | 表名必须以合法分层标识开头：`ods_`、`dim_`、`dwd_`、`dws_`、`ads_` | 高 |
| L-02 | 不允许出现模糊或临时前缀，如 `tmp_`、`test_`、`bak_`、`old_`、`dw_` | 中 |
| L-03 | 分层前缀只能出现一次，不允许叠加写法，如 `ods_dwd_xxx` | 中 |

---

### 维度二：各层命名结构

> 每层结构规则不同，检查时须先识别层次，再对照对应层的规则。

#### ODS 层

**格式**：`ods_{源系统名称}_{数据库名}_{源系统表名}`

- 段数要求：去掉 `ods_` 前缀后，必须恰好包含三段（源系统、数据库、表名），不多不少。
- 各段含义不允许缺省，不允许合并。

| 规则 ID | 规则描述 | 严重级别 |
|--------|---------|---------|
| S-ODS-01 | 必须包含源系统名称、数据库名、源系统表名三段 | 高 |
| S-ODS-02 | 不允许省略数据库名直接写 `ods_{源系统}_{表名}` | 高 |

**合规示例**：`ods_uf_ha_client`（源系统=uf，数据库=ha，表名=client）

---

#### DIM 层

**格式**：`dim_{业务板块缩写/pub}_{维度定义}[_{自定义命名标签}]`

- `pub` 用于与具体业务板块无关、或各业务板块可公用的维度（如时间、地区）。
- 业务板块缩写由团队统一定义，检查时如发现未知缩写，标注"请确认业务板块缩写是否已注册"。

| 规则 ID | 规则描述 | 严重级别 |
|--------|---------|---------|
| S-DIM-01 | 去掉 `dim_` 后，第一段必须是业务板块缩写或 `pub` | 高 |
| S-DIM-02 | 第二段维度定义不能缺省，不允许 `dim_pub` 这种只有一段的写法 | 高 |
| S-DIM-03 | 自定义命名标签（第三段）为可选项，若有多个标签需合并为一段，不允许无限叠加 | 低 |

**合规示例**：
- `dim_pub_area`（公共区域维表）
- `dim_prd_sec_stockgroup`（产品域证券维表，按组/类别）

---

#### DWD 层

**格式**：`dwd_{业务板块缩写/pub}_{数据域缩写}_{业务过程缩写}[_{自定义表命名标签缩写}][_{增量全量标识}]`

- 增量全量标识：`i`（增量）、`f`（全量），实时表可省略。
- 前三段（业务板块、数据域、业务过程）为必填，后两段为可选。

| 规则 ID | 规则描述 | 严重级别 |
|--------|---------|---------|
| S-DWD-01 | 去掉 `dwd_` 后，必须包含业务板块、数据域、业务过程三段 | 高 |
| S-DWD-02 | 增量全量标识若出现，只允许 `i` 或 `f`，且必须放在最末段 | 中 |
| S-DWD-03 | 各段均应使用缩写而非全称，单段长度建议不超过 8 个字符 | 低 |

**合规示例**：`dwd_crdt_agr_quota_chg`（信贷业务板块，协议数据域，额度变更业务过程）

---

#### DWS 层

**格式**：`dws_{业务板块缩写/pub}_{主题域缩写}_{数据粒度缩写}[_{自定义表命名标签缩写}][_{统计时间周期缩写}]`

时间周期缩写规则：
- 缺省情况下离线计算应包含 `_1d`（最近一天）、`_nd`（最近N天）、`_td`（历史截至当天）三个表。
- `_nd` 拆分时只允许以单一统计周期为单元，如 `_1w`（最近7天），不允许一张表存多个周期。
- 小时表统一用 `_hh`，分钟表统一用 `_mm`，实时统计省略时间周期后缀。

| 规则 ID | 规则描述 | 严重级别 |
|--------|---------|---------|
| S-DWS-01 | 去掉 `dws_` 后，必须包含业务板块、主题域、数据粒度三段 | 高 |
| S-DWS-02 | 时间周期缩写若出现，必须符合 `_1d/_nd/_td/_1w/_hh/_mm` 规范，不允许自定义格式如 `_day`、`_7days` | 中 |
| S-DWS-03 | 不允许一张表的名称暗示包含多个统计周期，如 `_1d_7d` | 高 |

**合规示例**：`dws_crdt_ast_accnt_debt`（信贷业务，资产主题域，账户粒度，实时统计）

---

#### ADS 层

**格式**：参考 DWS 层命名规则，根据业务需要适当调整，灵活性相对更高。

- 必须保留 `ads_` 前缀 + 业务板块缩写的基本结构。
- 后续段落可根据报表/应用场景命名，但仍须遵守字符规范和语义清晰度要求。

| 规则 ID | 规则描述 | 严重级别 |
|--------|---------|---------|
| S-ADS-01 | 去掉 `ads_` 后，至少包含业务板块缩写和一段业务描述，不允许只有一段 | 高 |
| S-ADS-02 | 命名风格应与 DWS 层保持整体一致，不允许同项目中 DWS/ADS 混用完全不同的命名模式 | 低 |

**合规示例**：`ads_crdt_ast_accnt_monit`（信贷业务，资产账户监控）

---

### 维度三：字符规范

| 规则 ID | 规则描述 | 严重级别 |
|--------|---------|---------|
| C-01 | 只允许小写字母、数字、下划线，不允许大写字母 | 高 |
| C-02 | 不允许出现中文字符 | 高 |
| C-03 | 不允许以数字或下划线开头 | 高 |
| C-04 | 不允许以下划线结尾 | 中 |
| C-05 | 不允许出现连续两个及以上下划线，如 `__` | 中 |
| C-06 | 不允许使用 Hive 保留关键字作为表名（如 `table`、`select`、`join`、`where`） | 高 |
| C-07 | 全表名长度建议不超过 64 个字符 | 低 |

---

### 维度四：语义清晰度

此维度需结合上下文判断，属于建议性规则，信息不足时降级为"建议"。

| 规则 ID | 规则描述 | 严重级别 |
|--------|---------|---------|
| M-01 | 各段应使用有意义的英文缩写，不允许使用拼音（如 `yonghu` 应改为 `user`） | 中 |
| M-02 | 不允许使用过于宽泛或冗余的词，如单独使用 `info`、`data`、`detail_info` | 低 |
| M-03 | 同一业务域内同类对象命名应保持一致，不允许 `order` 与 `orders` 并存 | 中 |
| M-04 | 缩写应全项目统一，不允许 `amt` 与 `amount` 混用 | 低 |

---

### 维度五：表注释规范

表注释通过建表语句中的表级 `COMMENT` 子句提供，形如：

```sql
CREATE TABLE dwd_crdt_agr_quota_chg (...)
COMMENT '客户征信授信额度变更表';
```

| 规则 ID | 规则描述 | 严重级别 |
|--------|---------|---------|
| T-01 | 建表语句必须包含表级 `COMMENT`，不允许缺失 | 高 |
| T-02 | 注释不允许为空字符串或仅有空格 | 高 |
| T-03 | 注释不允许与表名直接转写相同，如表名 `dwd_trade_order` 注释写成 `dwd trade order`（无效注释） | 高 |
| T-04 | 注释应使用中文，清晰描述业务含义，不允许纯英文缩写堆砌 | 中 |
| T-05 | 注释应包含以下要素中的至少两项：①业务对象是什么、②记录了什么事件/状态、③数据粒度/范围 | 中 |
| T-06 | 注释长度建议在 10～50 个汉字之间：过短（< 5 字）说明不足，过长（> 80 字）应精简 | 低 |
| T-07 | 不允许出现"待补充"、"TODO"、"暂无"、"测试"等占位词 | 中 |

**合规注释参考示例**：

| 表名 | 合规注释 | 不合规示例及原因 |
|-----|---------|----------------|
| `dwd_crdt_agr_quota_chg` | `信贷业务协议层-客户授信额度变更明细表` | `dwd crdt agr`（纯缩写堆砌，违反 T-04）|
| `dws_crdt_ast_accnt_debt` | `信贷资产主题-账户维度信用资产负债实时汇总表` | `汇总表`（缺少业务对象和数据范围，违反 T-05）|
| `dim_pub_area` | `公共维度-行政区域层级维表，含省市区三级` | `area`（与表名转写无异，违反 T-03）|

---

### 维度六：模型冗余

此维度为跨表检查，需要对比同一文档内所有表的字段集合。

| 规则 ID | 规则描述 | 检测方法 | 严重级别 |
|---------|---------|---------|---------|
| R-01 | 同一层内两张表字段名重复度超过 60%，建议评估是否可合并 | 计算两表字段名的 Jaccard 相似度：交集大小 / 并集大小，> 0.6 且同属一个分层时告警 | 中 |
| R-02 | 一张表的字段是另一张表字段的真子集（A 表字段全部包含于 B 表），建议评估是否被覆盖或冗余 | 判断 A 表字段集合是否为 B 表字段集合的子集，且 B 表字段数明显多于 A 表 | 中 |
| R-03 | 同一业务域内存在两张表名相似度高但非版本/时间后缀差异的表，建议人工确认是否为重复建模 | 对比表名相似度（去掉分层前缀和业务域前缀后），> 0.7 且无 `_v2`/`_bak`/日期等合理后缀时告警 | 低 |

## 核心工作流

### Step 1：解析 DDL 文档，提取所有表

从完整 DDL 文档中逐一提取每张表的以下信息：
- **表名**：`CREATE TABLE` 后的名称，去掉数据库前缀（`db.table` 只取 `table`）
- **表注释**：表级 `COMMENT` 的值；若不存在则记录为"缺失"
- **字段集合**：表内所有字段名列表（用于维度六跨表比对）
- 去重，统计总表数

### Step 2：识别层次

根据表名前缀判断所属层次（`ods/dim/dwd/dws/ads`）。
- 前缀不在合法范围内 → 触发 L-01，同时标注"无法确定层次，跳过结构检查"。

### Step 3：逐表检查

对每张表依次执行表级检查（维度一～五），以及跨表检查（维度六）：
1. 维度一（分层前缀）：验证 L-01 ～ L-03
2. 维度二（命名结构）：根据已识别层次，对照对应层的规则
3. 维度三（字符规范）：验证 C-01 ～ C-07
4. 维度四（语义清晰度）：验证 M-01 ～ M-04，信息不足时标注"建议"
5. 维度五（表注释）：验证 T-01 ～ T-07
6. 维度六（模型冗余）：对文档内所有表两两比对，验证 R-01 ～ R-03

检查原则：发现问题后不短路，继续检查其余维度；同一张表可同时存在多个问题。

### Step 4：汇总问题，分级排序

```
🔴 高优先级 — 违反强制规范，必须整改
🟡 中优先级 — 影响可读性、一致性或注释质量，建议整改
🔵 低优先级 — 优化建议，可在下次迭代处理
```

### Step 5：生成 Excel 报告

## 注意事项

1. **数据库前缀处理**：表名带有库限定符（如 `ods_db.ods_uf_ha_client`）时，只检查表名部分。
2. **层次无法识别**：前缀不合规的表先触发 L-01，结构检查（维度二）标注"无法确定层次，跳过"，其余维度照常检查。
3. **拼音识别**：对疑似拼音的词（非常见英文词且符合汉语拼音规律），标注"疑似拼音，请人工确认"，不武断判定。
4. **缩写一致性**：同一文档内发现同义缩写不统一（如 `amt` 与 `amount` 并存），触发 M-04 并列举所有涉及表。
5. **大批量输入**：超过 50 张表时，高优先级问题逐条列出，中低优先级合并为统计摘要，避免 Excel 过长。
6. **英文注释团队**：若用户明确说明团队使用英文注释，T-04 降级为"建议"，不作为必须整改项。
7. **仅表级检查**：本 skill 不检查字段命名、字段类型、字段注释、字段设计等字段级内容，请使用配套的字段级 skill 进行完整审查。
