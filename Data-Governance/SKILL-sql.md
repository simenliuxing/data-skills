# 数据治理 · SQL 代码规范检查

## 概述

本 skill 专门对 Hive 数仓的 **SQL 代码** 进行静态规范性审查，不涉及外部元数据或运行时数据。
覆盖以下三个维度：

1. **安全合规**（SEC-01 ～ SEC-02）
2. **性能优化**（PERF-01 ～ PERF-06）
3. **代码规范**（CODE-01 ～ CODE-11）

输入为 SQL 代码文本（文件上传或直接粘贴均可）。
输出为 **Excel 文件**，包含「代码问题」和「检查摘要」两个 sheet 页。

> **边界说明**：本 skill 仅基于 SQL 静态文本进行检测。凡是需要外部元数据（如字段类型、数据量、敏感标签）或运行时信息才能判断的规则，已排除在外。

---

## 前置要求

收到输入后，无需提前询问，直接开始解析和检查。以下情况例外：

- **特殊豁免**：用户主动说明某些代码为历史遗留或临时脚本，可降级处理，记录豁免原因。
- **语法异常**：SQL 语法错误导致无法解析 AST 时，降级为文本正则匹配模式（仅做可文本检测的规则），并在报告中注明"降级检查，可能有遗漏"。
- **自定义分区字段名（可选）**：SEC-02 需要判断 DML 是否包含分区过滤，默认按 `dt`、`partition_date`、`p_date`、`month_id`、`day_id` 推断分区字段。如果你们项目使用了不同的分区字段名（如 `trade_date`、`biz_dt`），请在输入时一并告知，格式：`分区字段名：xxx`，否则非标命名的分区过滤会被漏检，导致 SEC-02 误报。

## 规则库

### 维度一：安全合规

此维度为 P0 级强制规则，违反即阻断。

| 规则 ID | 规则描述 | 检测方法 | 严重级别 |
|--------|---------|---------|---------|
| SEC-01 | SQL 中不允许硬编码明文密码、AK/SK、Token 等敏感凭证 | 文本匹配 `password=`、`access_key=`、`secret=`、`token=` 后面跟字符串常量 | 高 |
| SEC-02 | 生产 DML（`DELETE`、`UPDATE`、`TRUNCATE`）必须包含分区过滤条件 | 解析 WHERE 条件是否包含分区字段。默认按 `dt`、`partition_date`、`p_date`、`month_id`、`day_id` 推断；用户在前置要求中指定了自定义分区字段名时，以用户指定为准。非标命名且未声明时，检测结果标注"⚠️ 分区字段名未确认，可能存在漏检，请人工复核" | 高 |

---

### 维度二：性能优化

| 规则 ID | 规则描述 | 检测方法 | 严重级别 |
|--------|---------|---------|---------|
| PERF-01 | 不允许隐式笛卡尔积（FROM 多表但无 JOIN） | AST 解析：FROM 子句表数量 > 1 且 JOIN 数量为 0 | 高 |
| PERF-02 | ORDER BY 必须搭配 LIMIT，不允许对大结果集做全局排序 | AST 检测存在 ORDER BY 但无 LIMIT | 高 |
| PERF-03 | 非等值 JOIN 需附注释说明原因 | AST 解析 JOIN ON 条件含非等值操作符（`>`、`<`、`LIKE` 等），检查上方是否有注释 | 中 |
| PERF-04 | 子查询嵌套不超过 3 层，超过时建议使用 CTE 重构 | AST 解析子查询嵌套深度 | 中 |
| PERF-05 | 同一表在 SQL 中多次出现且均无 WHERE 过滤，建议使用中间结果复用 | AST 统计同一表名在 FROM/JOIN 中出现次数，且每次均无 WHERE 限制 | 中 |
| PERF-06 | 单条 SQL 中 JOIN 表数量不应超过 8 张，过多时应拆分或引入中间表 | 统计单条 SQL 中 JOIN 关键字数量 | 中 |

---

### 维度三：代码规范

#### 3.1 命名规范

| 规则 ID | 规则描述 | 检测方法 | 严重级别 |
|--------|---------|---------|---------|
| CODE-01 | 表别名必须有意义，不允许使用 `a`、`b`、`c`、`t` 等无意义单字母别名 | AST 解析 `table AS alias` 或 `table alias`，判断别名长度 | 中 |
| CODE-02 | CTE（WITH 语句）名称应具有业务含义，不允许 `t1`、`t2`、`tmp1` 等 | AST 解析 `WITH name AS`，判断名称是否有意义 | 中 |

#### 3.2 注释规范

| 规则 ID | 规则描述 | 检测方法 | 严重级别 |
|--------|---------|---------|---------|
| CODE-03 | SQL 开头必须包含注释块，说明脚本用途和作者/Owner | 文本检查 SQL 前 200 字符是否有 `--` 或 `/* */` 注释 | 中 |
| CODE-04 | `CASE WHEN` 分支超过 3 个时，上方应附注释说明业务含义 | 文本检测 `CASE WHEN` 分支数 > 3 且上方无注释 | 中 |
| CODE-05 | 注释关键词与实际代码行为不一致 | 文本匹配注释中的操作词（如"过滤"、"排除"、"包含"）与 WHERE 条件对比 | 高 |

#### 3.3 逻辑规范

| 规则 ID | 规则描述 | 检测方法 | 严重级别 |
|--------|---------|---------|---------|
| CODE-06 | WHERE 中不允许使用 OR 连接不同表的字段 | AST 解析 WHERE OR 条件涉及的字段来源表 | 中 |
| CODE-07 | JOIN 条件中不允许混用 WHERE 过滤条件，应严格分离 | AST 解析 JOIN ON 与 WHERE 的分离 | 中 |
| CODE-08 | `IN` 列表值超过 100 个时，建议改为临时表 JOIN | 文本计数 `IN (...)` 中的逗号数量 | 中 |
| CODE-09 | 窗口函数必须包含 `PARTITION BY` 和 `ORDER BY`，不允许省略 | AST 解析 `OVER()` 内容 | 高 |
| CODE-10 | 不允许裸写 `SELECT DISTINCT *` | 文本匹配 | 高 |
| CODE-11 | 跨时区时间转换必须显式指定时区参数 | AST 解析 `to_utc_timestamp`/`from_utc_timestamp` 的参数数量 | 中 |

## 核心工作流

### Step 1：解析 SQL 代码

从输入中提取每个 SQL 语句的以下信息：
- **SQL 文本**：完整的 SQL 语句
- **代码来源**：文件名或"直接粘贴"
- **涉及表清单**：FROM / JOIN 中引用的所有表
- **输出表**：INSERT INTO / CREATE TABLE 的目标表

统计总 SQL 数、总涉及表数。

### Step 2：逐条 SQL 检查

对每条 SQL 依次执行：

1. 维度一（安全合规）：验证 SEC-01、SEC-02
2. 维度二（性能优化）：验证 PERF-01 ～ PERF-06
3. 维度三（代码规范）：验证 CODE-01 ～ CODE-11

检查原则：发现问题后不短路，继续检查其余维度；同一条 SQL 可同时存在多个问题。

### Step 3：汇总问题，分级排序

```
🔴 高优先级 — 违反强制规范（安全/性能），必须整改
🟡 中优先级 — 影响可读性、一致性或代码质量，建议整改
🔵 低优先级 — 优化建议，可在下次迭代处理
```

### Step 4：生成 Excel 报告

使用 xlsx skill，生成包含 **「代码问题」** sheet 页的 Excel 文件。
Sheet 列定义如下：

| 列名 | 说明 |
|------|------|
| SQL 来源 | 文件名或"直接粘贴" |
| SQL 摘要 | SQL 前 100 字符摘要（过长截断） |
| 问题类型 | "安全合规" / "性能优化" / "代码规范" |
| 严重级别 | 高 / 中 / 低 |
| 规则 ID | 触发的规则编号，如 SEC-01、PERF-03 |
| 问题描述 | 具体的违规描述 |
| 整改建议 | 具体的修改建议 |
| 原因说明 | 简要解释为什么需要整改 |

## 注意事项

1. **输入形式**：支持 `.sql` 文件上传和直接粘贴 SQL 文本。多条 SQL 以 `;` 分隔自动拆分。
2. **AST 解析降级**：SQL 语法复杂或包含 Hive 方言扩展导致无法解析 AST 时，降级为文本正则匹配模式，在报告中注明"降级检查，可能有遗漏"。
3. **安全合规优先**：安全合规维度（SEC）为 P0 级，发现后在报告中置顶展示。
4. **分区字段推断**：SEC-02 默认按 `dt`、`partition_date`、`p_date`、`month_id`、`day_id` 推断分区字段。若项目使用非标命名（如 `trade_date`、`biz_dt`），须在输入时告知，否则 SEC-02 结果不可靠。当检测到输入中存在 DML 语句但 WHERE 中无任何常见分区字段时，报告中自动附注："未检测到标准分区字段，如项目使用自定义分区字段名请告知后重新检查"。
5. **大批量输入**：超过 50 条 SQL 时，高优先级问题逐条列出，中低优先级合并为统计摘要，避免 Excel 过长。
6. **历史遗留代码**：用户标注为历史代码时，CODE 类规则降级为"建议"，但 SEC 和 PERF 类规则仍强制执行。
7. **与 DDL 检查的关系**：本 skill 不检查建表语句的命名规范（表名/字段名/注释等），请使用配套的表级/字段级 DDL 规范检查 skill。如果输入中包含 CREATE TABLE 语句，仅检查其中的 SELECT/INSERT 逻辑部分，不检查列定义命名。
8. **提升 SEC-02 准确率的方式**：在提交 SQL 前，在文件顶部或输入消息中加一行注释说明分区字段，例如：`-- 本项目分区字段：biz_dt`，skill 将自动识别并替换默认推断字段列表。
