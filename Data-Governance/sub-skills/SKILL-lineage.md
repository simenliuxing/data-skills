# 数据治理 · 血缘健康检查

## 概述

本 skill 专门对 Hive 数仓 SQL 代码中的 **表级血缘关系** 进行静态审查，通过解析 SQL 中的 FROM/JOIN/INSERT INTO 等语句，构建表与表之间的依赖图谱，评估血缘健康度。
覆盖以下四个维度：

1. **血缘覆盖率**（CV-01 ～ CV-03）
2. **越层依赖**（J-01 ～ J-04）
3. **血缘完整性**（L-01 ～ L-02）
4. **血缘复杂度**（P-01 ～ P-04）

输入为包含一个或多个 SQL 语句的文本内容（文件上传或直接粘贴均可）。
输出为 **Excel 文件**，每个大类一个 sheet 页，上半部分是明细问题，下半部分是检查摘要。

> **边界说明**：本 skill 仅基于 SQL 静态文本构建血缘关系。ODS 层表作为源系统表，允许无上游。

---

## 前置要求

收到输入后，无需提前询问，直接开始解析和检查。以下情况例外：

- **特殊豁免**：用户主动说明某些代码为历史遗留或临时脚本，可降级处理，记录豁免原因。
- **语法异常**：SQL 语法错误导致无法解析 AST 时，降级为文本正则匹配模式（仅做可文本检测的规则），并在报告中注明"降级检查，可能有遗漏"。

---

## 规则库

### 维度一：血缘覆盖率

能否解析出完整的表级血缘链路。

| 规则 ID | 规则描述 | 检测方法 | 严重级别 |
|---------|---------|---------|---------|
| CV-01 | 非 ODS 层表应有明确的上游来源表（ODS 层表作为源系统表，允许无上游） | 解析 SQL 中 FROM/JOIN 引用的表，识别每张非 ODS 表是否有对应的 INSERT/SELECT 定义其加工逻辑，统计未解析上游的表占比 | 高 |
| CV-02 | 每张被加工的表应有下游消费表，不允许"加工完没人用"的孤立表 | 反向追溯：所有被 INSERT INTO 的目标表，检查是否在其他 SQL 中被 FROM/JOIN 引用 | 中 |
| CV-03 | 血缘解析覆盖率应 ≥ 90%，未解析的表需要人工补充说明 | 未解析表数 / 总表数（不含 ODS 层源表） | 中 |

---

### 维度二：越层依赖

表与表之间的依赖是否符合数据分层逻辑。

| 规则 ID | 规则描述 | 检测方法 | 严重级别 |
|---------|---------|---------|---------|
| J-01 | 不允许跳过中间层直接跨层依赖，如 ADS 直接读 ODS 表 | 解析 SQL 中目标表与 FROM/JOIN 源表的层前缀，判断中间层是否被跳过（如 ads → ods 跳过 dwd/dws） | 高 |
| J-02 | 不允许同层表互相依赖，如 DWD 表 A 的加工逻辑依赖同层 DWD 表 B（同层表应各自独立加工自上游） | 解析 SQL 中源表和目标表的层前缀，判断是否同层 | 中 |
| J-03 | 不允许反向依赖，如 ODS 层引用 DWD/DWS/ADS 层表 | 解析 SQL 中 FROM/JOIN 表名前缀，判断是否从低层引用高层表 | 高 |
| J-04 | 跨业务域依赖需要注释说明原因，如信贷域表直接引用交易域表 | 解析 SQL 中引用表的业务域前缀（表名第二段）与目标表业务域是否一致，不一致且无注释说明则告警 | 低 |

---

### 维度三：血缘完整性

血缘链路是否闭环、是否有断点。

| 规则 ID | 规则描述 | 检测方法 | 严重级别 |
|---------|---------|---------|---------|
| L-01 | 从源系统到 ADS 的血缘链路应连续，不允许中间出现断链 | 从 ADS 层表向上追溯依赖链路，检查是否能追溯到 ODS 层；断链时标注断裂位置 | 高 |
| L-02 | 一张表被多个不同任务的 SQL 写入时需预警（如被多条独立 SQL 同时 INSERT INTO），说明存在写入冲突风险 | 统计 INSERT INTO / CREATE TABLE AS 的目标表，同一表被不同 SQL 语句作为目标时告警 | 高 |

---

### 维度四：血缘复杂度

血缘链路是否过于复杂，影响可维护性。

| 规则 ID | 规则描述 | 检测方法 | 严重级别 |
|---------|---------|---------|---------|
| P-01 | 单表上游来源表不应超过 10 张，过多说明加工逻辑过于复杂 | 统计单条 SQL 中 FROM/JOIN 的表数量 | 中 |
| P-02 | 单表不应被超过 20 张下游表引用，说明粒度过细或被滥用 | 统计下游引用数（该表在其他 SQL 中被 FROM/JOIN 的次数） | 低 |
| P-03 | 血缘链路深度（ODS 到 ADS 的跳转层数）不应超过 4 层 | 从 ODS 到 ADS 的依赖图谱中 BFS/DFS 最长路径长度 | 中 |
| P-04 | 不允许"扇入+扇出"双高（一张表同时有大量上游和下游），说明该表承担不合理的中转角色 | 上游数 > 10 且下游数 > 10 | 低 |

## 核心工作流

### Step 1：解析 SQL，构建血缘图谱

从输入中提取每个 SQL 语句的以下信息：
- **SQL 文本**：完整的 SQL 语句
- **代码来源**：文件名或"直接粘贴"
- **输出表**：INSERT INTO / CREATE TABLE AS 的目标表
- **上游表**：FROM / JOIN 中引用的所有表

构建表级血缘关系图谱（目标表 → [上游表列表]），统计：
- 总表数（去重后所有涉及的表）
- 总 SQL 数
- 血缘关系数（有明确加工逻辑的表数量）

### Step 2：逐维度检查

对血缘图谱依次执行：

1. 维度一（血缘覆盖率）：验证 CV-01 ～ CV-03
2. 维度二（越层依赖）：验证 J-01 ～ J-04
3. 维度三（血缘完整性）：验证 L-01 ～ L-02
4. 维度四（血缘复杂度）：验证 P-01 ～ P-04

检查原则：发现问题后不短路，继续检查其余维度。

### Step 3：汇总问题，分级排序

```
🔴 高优先级 — 血缘断裂、越层依赖、多任务写入冲突，必须整改
🟡 中优先级 — 上游过多、链路过深、同层依赖，建议整改
🔵 低优先级 — 下游滥用、扇入扇出双高，可在下次迭代处理
```

使用 xlsx skill 生成 Excel 文件，本类检查的结果放在一个独立的 sheet 页中。

## 注意事项

1. **输入形式**：支持 `.sql` 文件上传和直接粘贴 SQL 文本。多条 SQL 以 `;` 分隔自动拆分。
2. **AST 解析降级**：SQL 语法复杂或包含 Hive 方言扩展导致无法解析 AST 时，降级为文本正则匹配模式，在报告中注明"降级检查，可能有遗漏"。
3. **ODS 层豁免**：ODS 层表作为源系统表，允许无上游来源，不触发 CV-01。
4. **分层识别**：根据表名前缀（ods_/dwd_/dws_/ads_/dim_）判断所属层。前缀不合法的表标注"未知层"，跳过越层依赖检查但保留覆盖率和完整性检查。
5. **业务域识别**：根据表名前缀后的第二段（如 `dwd_crdt_xxx` 中的 `crdt`）判断业务域。
6. **大批量输入**：超过 50 条 SQL 时，高优先级问题逐条列出，中低优先级合并为统计摘要，避免 Excel 过长。
7. **历史遗留代码**：用户标注为历史代码时，越层依赖降级为"建议"，但血缘断裂和多任务写入规则仍强制执行。
8. **与 DDL/SQL 检查的关系**：本 skill 不检查建表语句的命名规范（表名/字段名/注释等）和 SQL 写法规范（安全/性能/格式），请使用配套的表级 DDL 规范检查（[SKILL-table.md](SKILL-table.md)）、字段级 DDL 规范检查（[SKILL-field.md](SKILL-field.md)）和 SQL 代码规范检查（[SKILL-sql.md](SKILL-sql.md)）skill。
9. **临时表处理**：以 `tmp_`、`tmp__` 开头的临时表不参与血缘分析，不计入覆盖率统计。
