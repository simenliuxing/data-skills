# 数据治理 · 字段级 DDL 规范检查

## 概述

本 skill 专门对 Hive 数仓建表语句的 **字段级别** 规范性进行审查，不涉及表级层面的检查。
覆盖以下四个维度：

1. **字段命名规范**（F-01 ～ F-08）
2. **字段类型合理性**（FT-01 ～ FT-07）
3. **字段注释规范**（FC-01 ～ FC-08）
4. **字段设计合理性**（FD-01 ～ FD-05）

输入为包含一个或多个 `CREATE TABLE` 语句的 DDL 文档。
输出为 **Excel 文件**，每个大类一个 sheet 页，上半部分是明细问题，下半部分是检查摘要。

> **边界说明**：本 skill 仅做字段级检查。表级检查请使用配套的「数据治理-表级DDL规范检查」skill（[SKILL-table.md](SKILL-table.md)）。如需检查字段是否与团队标准字段库绑定，请使用配套的「数据治理-标准字段绑定检查」skill（[SKILL-standard.md](SKILL-standard.md)）。

---

## 前置要求

收到输入后，无需提前询问，直接开始解析和检查。以下情况例外：

- **无法解析字段**：DDL 语句语法异常导致无法提取字段信息，向用户确认是否为截断内容。
- **特殊豁免**：用户主动说明某些表为历史遗留表，字段可降级处理，记录豁免原因。

## 规则库

### 维度一：字段命名规范

字段命名整体遵循与表名相同的字符规范（小写字母+数字+下划线），另有以下字段专属规则：

| 规则 ID | 规则描述 | 严重级别 |
|--------|---------|---------|
| F-01 | 只允许小写字母、数字、下划线，不允许大写字母或中文 | 高 |
| F-02 | 不允许使用拼音（如 `jine` 应改为 `amount`） | 中 |
| F-03 | 不允许与 Hive 保留关键字同名（如 `date`、`value`、`table`、`year`、`month`） | 高 |
| F-04 | 布尔/标志类字段必须使用 `is_` 前缀，如 `is_valid`、`is_deleted` | 中 |
| F-05 | 时间类字段必须以 `_time`、`_dt`、`_date`、`_ts` 结尾，如 `create_time`、`trade_dt` | 中 |
| F-06 | 金额类字段必须以 `_amt` 或 `_amount` 结尾，且同一文档内必须二选一统一使用 | 中 |
| F-07 | 同一张表内不允许出现含义相同或高度相似的字段名，如 `user_id` 和 `userid` 并存 | 高 |
| F-08 | 字段名长度建议不超过 40 个字符 | 低 |

---

### 维度二：字段类型合理性

| 规则 ID | 规则描述 | 严重级别 |
|--------|---------|---------|
| FT-01 | 金额/价格字段必须使用 `DECIMAL(精度, 小数位)`，不允许使用 `FLOAT` 或 `DOUBLE` | 高 |
| FT-02 | `DECIMAL` 类型必须显式指定精度和小数位，不允许裸写 `DECIMAL` | 高 |
| FT-03 | 日期字段应使用 `DATE` 或 `TIMESTAMP`，不允许用 `STRING` 存储日期 | 高 |
| FT-04 | 纯数值字段（如年龄、数量、编码）不允许使用 `STRING` 类型 | 中 |
| FT-05 | 枚举/状态字段推荐使用 `STRING`（配合字段注释说明值域），不建议使用 `INT` 存枚举码而注释缺失 | 中 |
| FT-06 | 同一文档中语义相同的 ID 字段类型必须一致，如 `client_id` 在不同表中不允许一处 `STRING` 一处 `BIGINT` | 中 |
| FT-07 | 不允许使用已过时或低效类型，如 `TINYINT`/`SMALLINT` 用于一般 ID、`BINARY` 存文本 | 低 |

---

### 维度三：字段注释规范

Hive 字段注释通过 `COMMENT '...'` 紧跟在字段定义后提供，形如：

```sql
client_id STRING COMMENT '客户唯一标识，关联 ods_uf_ha_client.client_id',
trade_status STRING COMMENT '交易状态：S-成功 F-失败 P-处理中',
trade_amt DECIMAL(18,2) COMMENT '交易金额，单位：元'
```

| 规则 ID | 规则描述 | 严重级别 |
|--------|---------|---------|
| FC-01 | 每个字段必须有 `COMMENT`，不允许缺失 | 高 |
| FC-02 | 注释不允许为空字符串、空格，或含"待补充"、"TODO"、"暂无"等占位词 | 高 |
| FC-03 | 注释不允许与字段名直接转写相同，如字段 `client_id` 注释写 `client id`（无效注释） | 高 |
| FC-04 | 枚举/状态类字段的注释必须列出所有合法值及其含义，格式建议：`状态：S-成功 F-失败` | 高 |
| FC-05 | 金额/数量类字段的注释必须说明单位，如"单位：元"、"单位：万元"、"单位：笔" | 中 |
| FC-06 | 外键/关联字段的注释应说明关联的表和字段，如"关联 dwd_crdt_agr.agr_id" | 中 |
| FC-07 | 注释应使用中文，不允许纯英文缩写堆砌 | 中 |
| FC-08 | 注释长度建议在 5～60 个汉字之间，过短说明不足，过长应精简 | 低 |

---

### 维度四：字段设计合理性

此维度基于单表 DDL 可判断的范围，不做跨表深度比对。

| 规则 ID | 规则描述 | 严重级别 |
|--------|---------|---------|
| FD-01 | 同一张表内不允许出现明显语义重叠的字段，如同时存在 `remark` 和 `memo` 且注释含义相同 | 中 |
| FD-02 | 不允许出现无限扩展式字段，如 `ext_info STRING`、`extra_col1`/`extra_col2` 等，这类字段意味着表设计需要重审 | 中 |
| FD-03 | DWD 层及以上的表建议包含 `create_time` 或 `insert_time` 字段用于数据追溯，缺失时给出建议 | 低 |
| FD-04 | 指标/度量类字段（金额、数量、比率）应在字段名或注释中明确统计口径，不允许仅写 `amt` 而无任何说明 | 中 |
| FD-05 | 单表字段数超过 60 个时，建议说明是否可拆分为宽表+扩展表结构，避免维护困难 | 低 |

## 核心工作流

### Step 1：解析 DDL 文档，提取所有表和字段

从完整 DDL 文档中逐一提取每张表的以下信息：
- **表名**：`CREATE TABLE` 后的名称，去掉数据库前缀（`db.table` 只取 `table`）
- **表所属层次**：根据表名前缀判断（`ods/dim/dwd/dws/ads`），用于 FD-03 检查
- **表注释**：表级 `COMMENT` 的值（辅助判断）
- **字段列表**：每个字段的名称、类型、注释

统计总表数和总字段数。

### Step 2：逐表逐字段检查

对每张表的每个字段依次执行：

1. 维度一（字段命名）：验证 F-01 ～ F-08
   - F-06 需在全表字段范围内判断 `_amt`/`_amount` 是否混用
2. 维度二（字段类型）：验证 FT-01 ～ FT-07
   - FT-06 需在当前文档范围内判断同名 ID 字段类型是否一致
3. 维度三（字段注释）：验证 FC-01 ～ FC-08
   - FC-04 需判断该字段是否属于枚举/状态类
4. 维度四（字段设计）：验证 FD-01 ～ FD-05
   - FD-03 需判断表所属层次是否为 DWD 及以上
   - FD-05 需统计当前表字段总数

检查原则：发现问题后不短路，继续检查其余维度；同一字段可同时存在多个问题。

### Step 3：汇总问题，分级排序

```
🔴 高优先级 — 违反强制规范，必须整改
🟡 中优先级 — 影响可读性、一致性或注释质量，建议整改
🔵 低优先级 — 优化建议，可在下次迭代处理
```

## 注意事项

1. **枚举字段识别**：字段名含 `status`、`type`、`flag`、`state`、`code` 等词，或类型为 INT/TINYINT 且无明显数值语义时，视为枚举/状态字段，触发 FC-04 检查值域说明。
2. **金额字段识别**：字段名含 `amt`、`amount`、`price`、`fee`、`cost`、`revenue` 等词时，触发 FT-01 检查类型是否为 DECIMAL，以及 FC-05 检查注释是否说明单位。
3. **时间字段识别**：字段名含 `time`、`dt`、`date`、`ts`、`timestamp` 等词时，触发 F-05 检查命名结尾，以及 FT-03 检查类型。
4. **拼音识别**：对疑似拼音的词（非常见英文词且符合汉语拼音规律），标注"疑似拼音，请人工确认"，不武断判定。
5. **缩写一致性**：同一文档内发现同义缩写不统一（如 `_amt` 与 `_amount` 并存），触发 F-06 并列举所有涉及表和字段。
6. **大批量输入**：超过 50 张表时，高优先级问题逐条列出，中低优先级合并为统计摘要，避免 Excel 过长。
7. **英文注释团队**：若用户明确说明团队使用英文注释，FC-07 降级为"建议"，不作为必须整改项。
8. **仅字段级检查**：本 skill 不检查表命名、表注释、分层前缀等表级内容，请使用配套的表级 skill 进行完整审查。
9. **标准字段绑定**：如需基于标准字段库检查字段绑定、类型一致性、注释一致性，请使用配套的「数据治理-标准字段绑定检查」skill（SKILL-standard.md）。两 skill 可组合使用：先做标准绑定，再做规范检查。
9. **跨表类型一致性**：FT-06 需要在全文档范围内比对同名 ID 字段的类型是否一致，如 `client_id` 在表 A 中为 `STRING` 而在表 B 中为 `BIGINT`，则在两张表中同时标注问题。
