# 数据治理 · 标准字段绑定检查

## 概述

本 skill 基于团队维护的 **标准字段库**，对 Hive 数仓建表语句（DDL）中的字段进行对照检查，判断字段是否已绑定标准、类型是否一致、注释是否匹配。

覆盖以下三个维度：

1. **标准绑定**（STD-01 ～ STD-02）
2. **类型一致性**（T-01 ～ T-02）
3. **注释一致性**（C-01 ～ C-02）

输入为：
- **标准字段库文件**：CSV 或 Excel 格式
- **DDL 文档**：包含一个或多个 `CREATE TABLE` 语句

输出为 **Excel 文件**，每个大类一个 sheet 页，上半部分是明细问题，下半部分是检查摘要。

> **边界说明**：本 skill 仅做字段与标准库的比对。字段命名规范、字段类型合理性、字段设计合理性等请使用配套的「数据治理-字段级DDL规范检查」skill（[SKILL-field.md](SKILL-field.md)）。

## 前置要求

### 文件说明

用户需提供以下文件之一（或两个都提供）：

| 文件 | 是否必须 | 说明 |
|------|---------|------|
| 标准字段库 | 必须 | CSV/Excel 格式，定义团队的标准字段字典 |
| 豁免字段库 | 可选 | CSV/Excel 格式，定义不参与标准绑定的技术字段 |

如果用户只提供 DDL 未提供标准字段库，提示用户上传标准字段库文件后重新检查。

### 标准字段库格式

CSV 或 Excel 文件，包含以下列：

| 列名 | 是否必须 | 说明 | 示例 |
|------|---------|------|------|
| 英文全称 | 必须 | 字段的标准英文名称，作为主要匹配依据 | `client_id` |
| 英文缩写 | 可选 | 字段的标准缩写形式，用于缩写匹配 | `cid` |
| 中文解释 | 必须 | 字段的标准中文含义描述 | `客户唯一标识` |
| 字段类型 | 必须 | 字段的标准数据类型 | `STRING` |
| 业务域 | 可选 | 字段所属的业务域，用于缩小匹配范围 | `crdt` |

**标准字段库示例**：

```csv
英文全称,英文缩写,中文解释,字段类型,业务域
client_id,cid,客户唯一标识,STRING,crdt
trade_amount,amt,交易金额，单位：元,DECIMAL(18,2),crdt
trade_status,,交易状态：S-成功 F-失败 P-处理中,STRING,crdt
order_no,on,订单编号,STRING,ord
```

- 英文缩列为空表示该字段无标准缩写
- 业务域为空表示该字段全局通用，不限定业务域

### 豁免字段库格式

CSV 或 Excel 文件，包含以下列：

| 列名 | 是否必须 | 说明 | 示例 |
|------|---------|------|------|
| 字段名 | 必须 | 不参与标准绑定的字段名 | `create_time` |
| 豁免原因 | 可选 | 说明为什么豁免该字段 | `技术元数据字段` |

**豁免字段库示例**：

```csv
字段名,豁免原因
create_time,系统自动生成的记录创建时间
insert_time,系统自动生成的记录插入时间
update_time,系统自动生成的记录更新时间
etl_dt,ETL 批次日期
etl_time,ETL 执行时间
is_deleted,逻辑删除标记
row_id,数据库行级唯一标识
partition_dt,分区日期
batch_no,ETL 批次编号
```

如果用户未提供豁免字段库，默认使用内置豁免清单：

```
create_time, insert_time, update_time, etl_dt, etl_time,
is_deleted, row_id, partition_dt, batch_no
```

## 规则库

### 维度一：标准绑定

| 规则 ID | 规则描述 | 严重级别 |
|--------|---------|---------|
| STD-01 | DDL 中的业务字段名必须在标准字段库的"英文全称"或"英文缩写"中至少匹配一项，否则触发"未绑定标准字段" | 中 |
| STD-02 | DDL 中的字段名同时匹配到标准字段库中多条记录，触发"匹配歧义" | 低 |

### 维度二：类型一致性

| 规则 ID | 规则描述 | 严重级别 |
|--------|---------|---------|
| T-01 | 已匹配到标准库的字段，DDL 定义的数据类型必须与标准库"字段类型"完全一致，否则触发"类型不一致" | 高 |
| T-02 | DECIMAL 类型字段精度不一致时需告警（如标准为 `DECIMAL(18,2)`，DDL 为 `DECIMAL(10,2)`） | 中 |

### 维度三：注释一致性

| 规则 ID | 规则描述 | 严重级别 |
|--------|---------|---------|
| C-01 | 已匹配到标准库的字段，DDL 的 COMMENT 内容必须与标准库"中文解释"语义一致（允许少量措辞差异，不允许缺失关键要素），否则触发"注释不一致" | 中 |
| C-02 | 已匹配到标准库的字段，DDL 中缺失 COMMENT 时触发"注释缺失" | 高 |

## 核心工作流

### Step 1：加载标准字段库和豁免字段库

1. 解析标准字段库文件，提取每条标准字段记录
2. 解析豁免字段库文件，如无则使用内置豁免清单
3. 构建标准字段索引（英文全称 + 英文缩写 → 标准记录）

### Step 2：解析 DDL，提取所有表和字段

从完整 DDL 文档中逐一提取每张表的以下信息：
- **表名**：`CREATE TABLE` 后的名称
- **表所属业务域**：根据表名前缀后的第二段判断（如 `dwd_crdt_xxx` 中的 `crdt`），用于标准匹配时缩小范围
- **字段列表**：每个字段的名称、类型、注释

统计总表数和总字段数。

### Step 3：逐字段匹配标准库

对每张表的每个字段依次执行：

1. **豁免检查**：判断字段是否在豁免字段库中
   - 是 → 跳过后续标准库检查，标记"已豁免"
   - 否 → 继续

2. **标准匹配**（STD-01）：
   - 如果表有业务域，先在标准库的业务域范围内匹配，未匹配再全局匹配
   - 按以下顺序匹配：
     a. 字段名 == 标准库"英文全称" → 精确匹配
     b. 字段名 == 标准库"英文缩写"（非空） → 缩写匹配
   - 未匹配到任何记录 → STD-01（未绑定标准字段）
   - 匹配到多条记录 → STD-02（匹配歧义），选择第一条继续后续检查

3. **类型检查**（T-01、T-02）：
   - 已匹配的字段，比较 DDL 类型与标准类型
   - 类型不同 → T-01（类型不一致）
   - DECIMAL 精度不同 → T-02（精度不一致）

4. **注释检查**（C-01、C-02）：
   - 字段无 COMMENT → C-02（注释缺失）
   - 字段有 COMMENT，但与标准库"中文解释"语义不一致 → C-01（注释不一致）
   - 注释一致性判断规则：
     - 标准注释中的核心名词必须出现在 DDL 注释中
     - 枚举类字段的值域描述（如 `S-成功 F-失败`）必须完整保留
     - 允许措辞微调，但不允许遗漏关键信息

检查原则：发现问题后不短路，继续检查其余维度；同一字段可同时存在多个问题。

### Step 4：汇总问题，分级排序

```
🔴 高优先级 — 类型不一致、标准字段注释缺失，必须整改
🟡 中优先级 — 未绑定标准、注释不一致、精度不一致，建议整改
🔵 低优先级 — 匹配歧义，可在下次迭代处理
```

## 注意事项

1. **输入形式**：标准字段库支持 `.csv` 和 `.xlsx` 格式上传。DDL 支持文件和直接粘贴。
2. **业务域匹配**：当 DDL 表名包含业务域前缀（如 `dwd_crdt_xxx` 中的 `crdt`）时，优先在标准库中匹配该业务域的字段，未匹配再扩大范围到全局。业务域为空的标准字段视为全局通用。
3. **类型比对**：类型比对按字符串完全匹配，`STRING` == `STRING`、`DECIMAL(18,2)` == `DECIMAL(18,2)`。大小写不敏感。
4. **注释一致性判断**：注释比对采用语义匹配而非字符串完全匹配。标准注释中的核心名词必须出现在 DDL 注释中；枚举字段的值域描述必须完整保留。允许措辞微调（如"客户 ID"与"客户唯一标识"视为一致）。
5. **豁免字段匹配**：豁免匹配区分大小写，按字段名精确匹配。内置豁免清单仅在用户未提供豁免字段库时使用。
6. **大批量输入**：超过 50 张表时，高优先级问题逐条列出，中低优先级合并为统计摘要，避免 Excel 过长。
7. **标准库字段缺失**：标准字段库中"英文全称"和"字段类型"必须有值，"中文解释"为空时跳过 C-01 和 C-02 检查。
8. **与 DDL 检查的关系**：本 skill 不检查字段命名规范（拼音、大小写、关键字等）、字段类型合理性、字段设计合理性等，请使用配套的字段级 DDL 规范检查 skill。两者可组合使用：先做标准绑定，再做规范检查。
